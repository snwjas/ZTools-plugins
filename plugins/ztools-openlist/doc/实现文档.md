# OpenListUI 实现文档

本文档面向后续想看懂、维护或改造本插件的人。项目是一个 ZTools 插件，前端使用 Vue 3 + TypeScript + Vite，UI 使用 Element Plus 按需导入，OpenList 相关网络请求和本地文件读写放在 preload 脚本里完成。

## 1. 项目结构

```text
my-first-plugin/
├─ public/
│  ├─ plugin.json              # ZTools 插件清单
│  ├─ logo.png                 # 插件图标和侧栏 Logo
│  └─ preload/
│     └─ services.js           # preload 服务，负责 OpenList API、本地读写、下载/上传进度
├─ src/
│  ├─ App.vue                  # ZTools 环境检测和入口路由
│  ├─ main.ts                  # Vue 挂载入口
│  ├─ OpenList/
│  │  └─ index.vue             # 主界面和业务逻辑
│  ├─ components/
│  │  └─ OperationFeedback.vue # 统一提示、确认弹窗、上传/下载进度
│  └─ env.d.ts                 # window.services 类型声明
├─ vite.config.js              # Element Plus 按需导入配置
└─ package.json
```

## 2. 运行与打包

开发运行：

```powershell
npm run dev
```

ZTools 开发模式会读取 `public/plugin.json` 里的：

```json
{
  "development": {
    "main": "http://127.0.0.1:5173"
  },
  "preload": "preload/services.js"
}
```

注意：浏览器直接打开 `http://127.0.0.1:5173` 时没有 ZTools 的 `window.ztools` 和 preload 注入，因此会显示“请在 ZTools 插件环境中运行”。必须从 ZTools 插件环境打开，文件选择、上传、下载、本地配置读写才可用。

生产打包：

```powershell
npm run build
```

`npm run build` 会执行两步：

1. `vue-tsc && vite build` 构建 Vue 前端，并把 `public/` 里的插件资源复制到 `dist/`。
2. `npm run build:preload` 使用 esbuild 单独打包 `public/preload/services.js`，输出到 `dist/preload/services.js`。

preload 脚本运行在 ZTools/Electron 的 CommonJS 环境里，不会被 Vite 前端 bundle 自动处理。由于请求层使用了 `axios`，生产包必须通过 `build:preload` 把 axios 一起打进 `dist/preload/services.js`，避免插件安装后出现 `Cannot find module 'axios'`。

ZTools 只加载一个 zip 时，应打包 `dist/` 目录内的产物，而不是打包项目源码目录。`dist/plugin.json`、`dist/preload/services.js`、`dist/logo.png`、`dist/index.html` 和 `dist/assets/` 都需要在 zip 根目录。

当前插件清单位于 `public/plugin.json`：

```json
{

  "name": "ztools-openlist",
  "title": "OpenListUI",
  "description": "在 ZTools 中浏览、下载和上传 OpenList 文件。",
  "author": "gcnanmu",
  "version": "1.0.0",
  "main": "index.html",
  "preload": "preload/services.js",
  "logo": "logo.png",
  "development": {
    "main": "http://127.0.0.1:5173"
  },
  "features": [
    {
      "code": "OpenListUI",
      "explain": "打开 OpenListUI 文件管理器",
      "icon": "logo.png",
      "cmds": ["openlist", "OpenList", "OpenListUI"]
    }

```

关键点：

- `preload` 指向 `preload/services.js`，ZTools 会把它注入到插件运行环境。

- `features[0].code` 是进入插件后的功能码，`App.vue` 会根据这个 code 显示 OpenList 界面。
- `logo.png` 同时作为插件图标和侧栏 Logo。

## 4. OpenList API 对接说明

所有 OpenList API 都通过 `public/preload/services.js` 发起，前端只调用 `window.services`。这样做的原因是上传、下载、本地配置持久化都需要 Node 能力，放在 Vue 渲染进程里不合适。

### 4.1 登录

URL：

```text
POST /api/auth/login
```

请求头：

```json
{
  "Content-Type": "application/json"
}
```

请求参数：

```json
{
  "username": "用户名",
  "password": "密码"
}
```

返回格式：

```json
{
  "code": 200,
  "message": "success",
  "data": {
    "token": "OpenList token"
  }
}
```

插件处理逻辑：

- 读取 `data.token`。
- 如果没有 token，抛出 `OpenList did not return a token`。
- 登录成功后保存 `{ baseUrl, username, token }` 到 ZTools userData 下的 `openlist-config.json`。

### 4.2 列出目录

URL：

```text
POST /api/fs/list
```

请求头：

```json
{
  "Authorization": "token",
  "Content-Type": "application/json"
}
```

请求参数：

```json
{
  "path": "/当前目录",
  "password": "",
  "page": 1,
  "per_page": 0,
  "refresh": false
}
```

返回格式：

```json
{
  "code": 200,
  "message": "success",
  "data": {
    "content": [
      {
        "name": "文件名.txt",
        "size": 1234,
        "is_dir": false,
        "modified": "2026-06-06T12:00:00Z",
        "sign": "",
        "thumb": "",
        "type": 0
      }
    ],
    "total": 1,
    "readme": "",
    "write": true
  }
}
```

插件处理逻辑：

- `content` 写入 `files`。
- 前端本地排序：目录在前，名称升序。
- 前端本地分页：每页固定 10 个。
- 搜索时基于当前目录返回结果本地过滤。

- 分享按钮不再调用 OpenList 分享接口，而是复制 `raw_url` 对应的文件直链。

### 4.4 上传文件

URL：

```text
PUT /api/fs/put
```

请求头：

```json
{
  "Authorization": "token",
  "Content-Type": "application/octet-stream",
  "Content-Length": "文件字节数",
  "File-Path": "encodeURIComponent('/目录/文件名.txt')",
  "As-Task": "false"
}
```

请求参数：

请求体是文件二进制内容，不是 JSON。

返回格式：

```json
{
  "code": 200,
  "message": "success",
  "data": null
}
```

插件处理逻辑：

- 前端使用 Element Plus `el-upload` 拖拽选择文件，但 `auto-upload=false`。
- 用户点击“开始上传”后，逐个文件调用 preload。
- 如果 ZTools/Electron 能拿到 `File.path`，使用本地文件流上传。
- 如果拿不到 `File.path`，使用浏览器 `File.arrayBuffer()` 上传。
- preload 里通过 axios HTTP adapter 上传二进制数据，并回传上传进度。
- 上传后强制刷新当前目录。

### 4.7 下载文件

OpenList 不是直接通过固定 API 下载。流程是：

1. 调用 `POST /api/fs/get` 获取文件 `raw_url`。
2. 通过 `GET raw_url` 下载文件。

请求头：

```json
{
  "Authorization": "token"
}
```

返回格式：

返回二进制文件流。

插件处理逻辑：

- 调用 `window.ztools.showSaveDialog` 选择保存路径。
- 用户取消保存时直接退出，不进入下载。
- 前端点击下载后会通过 `getFileLinkDetail` 获取文件链接 `raw_url`，这条逻辑与分享共用，并会复用 `fileLinkCache` 中已有的链接；preload 只负责下载前端传入的文件链接。
- preload 使用 axios 下载远程文件；保存到本地文件时强制使用 Node HTTP adapter，确保 `responseType: 'stream'` 返回真实分块流。
- 普通下载只接受 Node stream 或 Web ReadableStream，并通过 `fs.createWriteStream` 写入本地文件；如果响应不是可流式读取的数据，会直接抛错，不再退回 Buffer 保存。
- 下载进度在 preload 内做 200ms 节流，完成时强制回传最终 100%，避免每个 chunk 都触发 Vue 高频渲染。
- 下载完成后调用 `window.ztools.shellShowItemInFolder(outputPath)`。

### 4.8 重命名、移动和复制

重命名使用 OpenList 文件系统接口：

```text
POST /api/fs/rename
```

请求参数：

```json
{
  "path": "/当前目录/旧名称.txt",
  "name": "新名称.txt"
}
```

移动和复制使用 OpenList 的 move/copy 统一参数结构：

```text
POST /api/fs/move
POST /api/fs/copy
```

请求参数：

```json
{
  "src_dir": "/source",
  "dst_dir": "/destination",
  "names": ["file1.txt", "folder1"]
}
```

文档确认 `/api/fs/copy` 的参数就是 `src_dir`、`dst_dir`、`names`，和移动接口一致。`Authorization` header 直接传 token，不加 `Bearer` 前缀。

复制的实现需要额外注意：

- 不要只相信 `/api/fs/copy` 返回 `code: 200`。部分 OpenList 存储后端会在底层执行 `COPY` 时失败，例如日志中出现 `COPY /path/file.pdf: 423` 或 `405`，但接口层不一定同步暴露为失败。
- 文件跨目录复制时，插件先调用 `/api/fs/copy`，随后用 `/api/fs/list` 强制刷新目标目录并确认目标文件真的出现。
- 如果目标目录没有出现目标文件，插件会自动兜底为“读取源文件内容 -> 上传到目标目录”：先用 `/api/fs/get` 获取 `raw_url`，再下载为 `Buffer`，最后用 `/api/fs/put` 上传。
- 文件夹复制不直接依赖 `/api/fs/copy`，因为底层存储不支持服务端 COPY 时会直接失败。插件改为递归复制：`list` 源文件夹、`mkdir` 目标子文件夹、对子文件继续递归、对普通文件执行下载再上传。
- 同目录复制会自动生成不冲突的新名称，例如 `文件.pdf` -> `文件 - 副本.pdf`，后续冲突会递增为 `文件 - 副本 2.pdf`。

复制速度取决于实际执行路径：

| 场景 | 执行方式 | 性能特征 |
| --- | --- | --- |
| 文件跨目录复制，且 OpenList 服务端 COPY 可用 | `/api/fs/copy` + `/api/fs/list` 校验 | 快，主要由服务端完成 |
| 文件跨目录复制，但服务端 COPY 报错或校验失败 | `/api/fs/get` 下载为 `Buffer` + `/api/fs/put` 上传 | 慢，相当于重新传输一遍文件 |
| 文件夹复制 | 递归 `list + mkdir + get + put` | 慢，目录越深、文件越多越明显 |
| 同目录复制 | 生成副本名后手动复制 | 慢，用于避免同名冲突 |

因此用户反馈“复制很慢”时，优先判断是否走到了手动复制路径。这个兜底是为了保证复制结果真实落盘；如果后续希望优先速度，可以改成只调用 `/api/fs/copy`，失败时提示“当前存储不支持服务端复制”，不再自动下载再上传。

## 5. preload 服务完整代码

文件：`public/preload/services.js`

这段是插件最核心的接口层。下面是 axios 请求层、二进制上传和流式下载的核心结构摘录；完整实现以 `public/preload/services.js` 为准。

```js
const fs = require('node:fs')
const path = require('node:path')
const axios = require('axios')

function normalizeBaseUrl(baseUrl) {
  return String(baseUrl || '').trim().replace(/\/+$/, '')
}

async function request(baseUrl, endpoint, options = {}) {
  const method = String(options.method || 'GET').toLowerCase()
  const headers = options.headers || {}
  const res = await axios.request({
    baseURL: normalizeBaseUrl(baseUrl),
    url: endpoint,
    method,
    headers,
    data: options.data,
    responseType: options.responseType || 'json',
    validateStatus: () => true,
    maxBodyLength: Infinity,
    maxContentLength: Infinity
  })

  return parseOpenListResponse(res)
}

function parseOpenListResponse(res) {
  const payload = res.data || null

  if (res.status < 200 || res.status >= 300) {
    throw new Error(getPayloadMessage(payload, `HTTP ${res.status}`))
  }

  if (payload && typeof payload === 'object' && payload.code !== undefined && payload.code !== 200) {
    throw new Error(getPayloadMessage(payload, `OpenList error: ${payload.code}`))
  }

  return payload
}

async function uploadOpenListBuffer(baseUrl, token, remoteDir, fileName, data, dataLength, onProgress) {
  const remotePath = path.posix.join(remoteDir || '/', fileName)
  const contentLength = Number(dataLength) || (Buffer.isBuffer(data) ? data.length : 0)

  const res = await axios.put(`${normalizeBaseUrl(baseUrl)}/api/fs/put`, data, {
    adapter: 'http',
    headers: authHeaders(token, {
      'Content-Type': 'application/octet-stream',
      'Content-Length': String(contentLength),
      'File-Path': encodeURIComponent(remotePath),
      'As-Task': 'false'
    }),
    validateStatus: () => true,
    maxBodyLength: Infinity,
    maxContentLength: Infinity,
    onUploadProgress(progress) {
      if (typeof onProgress !== 'function') return
      const loaded = progress.loaded || 0
      const total = progress.total || contentLength
      onProgress({
        loaded,
        total,
        percent: total ? Math.round((loaded / total) * 100) : 100
      })
    }
  })
  const payload = parseOpenListResponse(res)

  return {
    path: remotePath,
    fileName,
    data: payload?.data || null
  }
}

async function downloadUrlToFile(url, token, savePath, fallbackTotal = 0, onProgress) {
  const res = await axios.get(url, {
    adapter: 'http',
    headers: authHeaders(token, {
      'Accept-Encoding': 'identity'
    }),
    responseType: 'stream',
    decompress: false,
    validateStatus: () => true,
    maxBodyLength: Infinity,
    maxContentLength: Infinity
  })

  if (res.status < 200 || res.status >= 300) {
    throw new Error(`HTTP ${res.status}`)
  }

  const total = Number(getResponseHeader(res.headers, 'content-length')) || fallbackTotal || 0
  fs.mkdirSync(path.dirname(savePath), { recursive: true })

  if (res.data && typeof res.data.pipe === 'function' && typeof res.data.on === 'function') {
    await writeNodeStreamToFile(res.data, savePath, total, onProgress)
  } else if (res.data && typeof res.data.getReader === 'function') {
    await writeWebStreamToFile(res.data, savePath, total, onProgress)
  } else {
    throw new Error('下载响应不是可流式读取的数据，已拒绝使用 Buffer 保存大文件')
  }

  return savePath
}

window.services = {
  async listOpenList(baseUrl, token, dirPath, password = '', refresh = false) {
    const payload = await request(baseUrl, '/api/fs/list', {
      method: 'POST',
      headers: authHeaders(token, { 'Content-Type': 'application/json' }),
      data: {
        path: dirPath || '/',
        password,
        page: 1,
        per_page: 0,
        refresh
      }
    })

    return payload?.data || { content: [], total: 0 }
  }
}
```
实现细节：

- `normalizeBaseUrl` 会删除末尾 `/`，避免拼接 API 时出现双斜杠。
- `request` 基于 `axios.request` 统一处理 OpenList JSON 响应，如果 HTTP 非 2xx 或 `code !== 200` 会抛错。
- `authHeaders` 统一追加 `Authorization`。
- `uploadOpenListBuffer` 使用 `axios.put` 上传二进制文件，支持 `Buffer` 或 Node stream；本地路径上传会通过 `fs.createReadStream` 流式上传，避免大文件一次性读入内存。
- `downloadOpenListFile` 只接收前端传入的 `fileUrl` 作为下载链接；如果没有传入链接会直接报错，不再在 preload 内部调用 `/api/fs/get`。
- 前端通过和分享相同的 `/api/fs/get` 链路取得 `raw_url`，并缓存在 `fileLinkCache` 中，后续分享或下载同一个文件会复用该链接。
- 文件夹下载不使用 `/api/fs/add_offline_download`。该接口用于让 OpenList 服务端把外部 HTTP、磁力或种子链接离线下载到 OpenList 存储，不适合把 OpenList 文件夹保存到本机。
- `downloadOpenListDir` 会在 preload 中递归调用 `/api/fs/list` 获取文件夹内容，对每个普通文件调用 `/api/fs/get` 取得 `raw_url`，再用本地流式下载写入用户选择的目录。
- 多选下载通过 `downloadOpenListItems` 处理，前端只选择一次本地目录；选中的文件直接保存到该目录，选中的文件夹会作为子目录递归保存。
- 文件夹下载会保留远程目录结构，并对本地路径片段做非法字符替换，避免 Windows 路径字符导致写入失败。
- 文件夹下载进度优先使用所有文件的已知大小做字节聚合；如果存在未知大小文件，则退化为按已完成文件数统计。
- 下载进度在 preload 内部做 200ms 节流，避免每个 stream chunk 都触发 Vue 响应式更新；下载完成时会强制发送最终 `100%` 进度。
- 文件夹和多选下载的聚合进度使用增量累计。每个文件只保存上一次 loaded，当前回调用 `delta` 累加总进度，不再每个 chunk 都全量遍历所有文件进度。
- 保存到本地文件的下载请求强制指定 `adapter: 'http'`，避免 axios 在 ZTools/Electron 中走 fetch adapter 后一次性拿到完整 Buffer，导致进度条只在结束时跳到 100。
- `downloadUrlToFile` 和 `downloadUrlToBuffer` 都会发送 `Accept-Encoding: identity`，并设置 `decompress: false`，要求 OpenList 或后端存储返回原始文件流。
- 这样可以规避部分 `raw_url` 响应头错误导致的 `incorrect header check`。该错误通常是响应声明了 gzip/deflate 等压缩格式，但实际返回的是普通二进制文件，Node 在自动解压时抛出的 zlib 错误。
- `downloadUrlToFile` 的写入层只接受 Node stream 或 Web ReadableStream。如果响应不是可流式读取的数据，会直接抛错，普通下载不再退回 Buffer 保存，避免大文件卡死界面。
- `downloadUrlToBuffer` 使用 axios `responseType: 'stream'`，只用于复制兜底时把源文件读成 `Buffer` 再上传到目标目录；它会先检查 `content-length`，并在流式读取过程中累计校验单文件最大 100MB，超过限制会立即中断读取。
- `loadOpenListConfig` 和 `saveOpenListConfig` 使用 ZTools `userData` 目录持久化配置。
- `copyOpenListFiles` 不只转发 `/api/fs/copy`。它会先确认源目录中的选中项类型；文件夹直接走递归复制，文件跨目录复制会先尝试 OpenList 原生 copy，再用 `/api/fs/list` 校验目标目录是否真的出现文件。
- 原生 copy 抛错或校验失败时，preload 会执行手动复制：文件通过 `/api/fs/get` 拿 `raw_url` 下载为 `Buffer`，再用 `/api/fs/put` 上传；文件夹通过 `list + mkdir + put` 递归复制。
- 同目录复制通过 `getAvailableCopyName` 生成 ` - 副本` 名称，避免目标目录同名冲突。

```ts
function normalizeUploadFile(file: any): UploadQueueItem {
  const raw = file.raw || file
  const filePath = (raw as File & { path?: string })?.path || file.path || ''

  return {

    name: raw?.name || file.name || getLocalFileName(filePath),
    size: raw?.size || file.size || 0,

    path: filePath
  }
}

function syncUploadQueue(_: any, files: any[]) {

}

function removeUploadFile(_: any, files: any[]) {

}

function clearUploadQueue() {

  uploadQueue.value = []
  uploadRef.value?.clearFiles()

async function uploadFiles() {
  if (!isConnected.value) {
    showToast('warning', '请先连接 OpenList')
    loginDialog.value = true
    return
  }

  if (uploadQueue.value.length === 0) {
    showToast('warning', '请先选择要上传的文件')
    return
  }

  uploading.value = true
  transferProgress.value = 0
  transferStatus.value = ''
  error.value = ''

  try {
    const uploadedNames: string[] = []
    const queue = [...uploadQueue.value]

    for (let index = 0; index < queue.length; index += 1) {
      const localFile = queue[index]
      transferStatus.value = `正在上传 ${index + 1}/${queue.length}：${localFile.name}`
      const onProgress = (progress: { percent: number }) => {
        const fileBase = index / queue.length
        const filePart = (progress.percent || 0) / 100 / queue.length
        transferProgress.value = Math.min(100, Math.round((fileBase + filePart) * 100))
      }

      if (!localFile.path && !localFile.raw) {
        throw new Error(`无法读取文件：${localFile.name}`)
      }

      const result: any = localFile.path
        ? await window.services.uploadOpenListFile(
            baseUrl.value,
            token.value,
            currentPath.value,
            localFile.path,
            onProgress
          )
        : await window.services.uploadOpenListFileContent(
            baseUrl.value,
            token.value,
            currentPath.value,
            localFile.name,
            await localFile.raw.arrayBuffer(),
            onProgress
          )
      if (result?.fileName) {
        uploadedNames.push(result.fileName)
      }
    }

    await refreshList(currentPath.value, true)

    const visibleNames = new Set(files.value.map((file) => file.name))
    const missingNames = uploadedNames.filter((name) => !visibleNames.has(name))

    if (missingNames.length > 0) {
      await sleep(1000)
      await refreshList(currentPath.value, true)
    }

    transferProgress.value = 100
    transferStatus.value = '上传完成'
    uploadDialog.value = false
    uploadQueue.value = []
    uploadRef.value?.clearFiles()
    showToast('success', `已上传 ${queue.length} 个文件`)
  } catch (err) {
    setError(err)
  } finally {
    window.setTimeout(() => {
      uploading.value = false
      transferProgress.value = 0
      transferStatus.value = ''
    }, 600)
  }
}
```

实现细节：

- Element Plus 的 `el-upload` 只负责选择和维护文件列表，不自动上传。
- `syncUploadQueue` 和 `removeUploadFile` 保持 Element Plus 内部列表和业务队列一致。
- 有 `File.path` 时走本地文件流上传，避免大文件一次性读入内存。
- 没有 `File.path` 时走 `arrayBuffer` 上传，兼容普通拖拽文件对象。
- 多文件上传进度按文件数量聚合。
- 上传弹窗中的 Element Plus 文件列表对长文件名做单行省略，避免选择超长文件名时撑开弹窗。

- 点击文件夹进入文件夹，不打开详情。
- 点击文件打开详情抽屉。
- 文件夹详情只通过右键菜单的“查看详情”进入。
- 移动到当前目录会被前端拦截。
- 移动文件夹到自身或子目录会被前端拦截。
- 复制到同目录允许执行，preload 会生成 ` - 副本` 名称。
- 多选状态下，移动和复制复用同一个目录树弹窗，但传给 preload 的 `names` 是所有选中项。

移动和复制弹窗使用 Element Plus `el-tree` 展示目录树。树数据不是一次性加载全量目录，而是在浏览目标目录时通过 `/api/fs/list` 加载当前目录和父目录的子目录，并保留：

- 上级
- 根目录
- 当前目录
- 刷新

这样移动/复制时既能回到上级目录，也能看到同级目录，并且避免只显示单一路径导致无法选择目标位置。

前端最终调用：

```ts
const action = mode === 'copy' ? window.services.copyOpenListFiles : window.services.moveOpenListFiles
await action(baseUrl.value, token.value, currentPath.value, dstDir, [file.name])
```

复制的可靠性不放在前端判断，而是在 preload 层实现。原因是 `/api/fs/copy` 的请求参数没有问题，但部分 OpenList 存储后端会返回“接口成功、底层 COPY 未落盘”的结果；preload 能读取文件内容并上传，因此更适合作为兜底层。

### 7.8 多选和批量操作

列表视图和图标视图都支持多选：

- 鼠标悬停到文件块或文件行时显示 Element Plus `el-checkbox`。
- 只要任意复选框被选中，就进入多选状态。
- 多选状态下所有文件块内的查看、下载、打开、分享、删除按钮隐藏，避免误触。
- 工具栏左侧常驻“全选/取消全选”按钮，按钮使用 Element Plus `el-button`；选择范围是当前页的 `pagedFiles`，不会跨页选择。
- 当前页已经全部选中时，再点“取消全选”只取消当前页，不会清除其他页已经选中的文件。
- 多选状态下，布局切换按钮左侧显示批量复制、批量移动、批量删除按钮，按钮使用 Element Plus `el-button-group` + `el-button`，不再使用自定义按钮外观。
- 点击已进入多选状态的文件块会切换该文件的选中状态，不会进入文件夹或打开详情。
- 刷新目录、切换目录或批量操作完成后会清空选中集合。
- 多选控件不手写 checkbox/button，本体使用 Element Plus；自定义 CSS 只控制复选框在 hover/多选状态下的显隐、图标视图里的定位，以及按钮组内部图标文字对齐。

暗色主题样式：

## 11. 排查问题

### 11.1 端口 5173 是否运行

```powershell
netstat -ano | Select-String ':5173'
```

也可以直接请求：

```powershell
Invoke-WebRequest -UseBasicParsing -Uri http://127.0.0.1:5173/ -TimeoutSec 3
```

### 11.2 页面提示“请在 ZTools 插件环境中运行”

原因通常是：

- 直接在浏览器打开了 Vite 页面，没有 `window.ztools`。
- ZTools 没有按插件方式打开。
- `plugin.json` 的 `preload` 路径错误。
- 打包 zip 时没有把 `preload/services.js` 放在 zip 根目录下的 `preload/` 目录中。

### 11.3 上传成功但列表不显示

当前代码上传成功后会：

1. `refreshList(currentPath.value, true)`
2. 检查上传文件名是否出现在列表中
3. 如果没出现，等待 1 秒后再次强刷

如果仍然不显示，重点检查：

- OpenList 服务端实际保存路径
- `File-Path` header 是否被服务端识别
- 当前目录是否有缓存
- 用户 token 是否有写权限

### 11.4 下载没有进度

下载进度优先依赖文件链接响应头 `content-length`，如果没有响应头，则使用列表接口返回的文件 `size` 作为兜底总量。保存到本地文件时 preload 会强制 axios 使用 Node HTTP adapter，让响应按 chunk 写入文件并持续回传进度；如果服务端和列表数据都不提供总大小，过程中仍只能显示未知总量，最终完成时变为 100。

### 11.5 点击下载后界面仍然卡死

当前已做的低风险优化包括：

- 普通下载强制使用 `responseType: 'stream'`，不再使用 `arraybuffer`。
- 普通下载只接受 Node stream 或 Web ReadableStream，不再退回 Buffer 保存。
- 本地保存使用 `fs.createWriteStream`，不使用 `fs.writeFileSync` 保存下载文件。
- 下载进度回调在 preload 内限制为 200ms 最多一次，完成时强制回调 100%。
- 文件夹和多选下载的聚合进度使用增量累计，不再每个 chunk 全量遍历所有文件进度。
- 复制兜底的 `downloadUrlToBuffer` 限制单文件最大 100MB，避免大文件整文件读入内存。

如果这些优化后仍然卡死，说明问题可能不在 Vue 高频渲染或 Buffer fallback，而在更底层的运行环境：

- ZTools 的 preload 可能仍与插件页面共享关键线程或事件循环，长时间网络流和本地 I/O 会影响界面响应。
- 文件夹下载在正式写文件前需要递归 `/api/fs/list`，并对每个普通文件调用 `/api/fs/get` 获取 `raw_url`，文件数量多时会有明显等待。
- 本地磁盘写入、杀毒软件扫描、网络盘路径或系统文件管理器联动也可能让保存阶段阻塞。
- 如果后续继续优化，应优先考虑把下载任务迁移到 ZTools/Electron main process、独立 worker 或独立下载任务进程，而不是继续在 preload 内堆逻辑。

### 11.6 复制提示成功但目标目录没有文件

优先确认请求体是否符合 OpenList 文档：

```json
{
  "src_dir": "/source",
  "dst_dir": "/destination",
  "names": ["file1.txt"]
}
```

如果请求体正确但仍然没有复制成功，通常不是前端参数问题，而是底层存储的服务端 COPY 不可用或异步失败。典型日志包括：

- `COPY /path/file.pdf: 423`
- `COPY /path/file.pdf: 405`

当前插件的修复策略：

1. 文件跨目录复制先调用 `/api/fs/copy`。
2. 调用后等待并强刷目标目录，确认目标文件名是否真的出现。
3. 如果没出现，自动改为 `/api/fs/get` 读取 `raw_url`，下载成 `Buffer`，再通过 `/api/fs/put` 上传到目标目录。
4. 文件夹复制直接走递归 `list + mkdir + put`，不依赖 `/api/fs/copy`。

如果后续仍然失败，重点检查：

- `/api/fs/get` 是否能返回源文件 `raw_url`
- 目标目录是否有写权限
- `/api/fs/put` 是否允许覆盖或创建文件
- 文件夹内是否包含无权读取的子项
- 目标目录是否已经存在同名文件或同名非文件夹项

## Gemini的修改建议

以下建议已审核并采纳到当前实现：

1. `downloadUrlToBuffer` 不再使用 `responseType: 'arraybuffer'` 一次性读取响应体，改为 `responseType: 'stream'`，先检查 `content-length`，并在读取过程中累计字节数，超过 100MB 立即中断流。
2. `writeNodeStreamToFile` 在网络流或本地文件写入流出错时，会同时销毁两端流，避免文件描述符、网络连接残留，以及半写入文件继续占用资源。
3. `uploadOpenListBuffer` 已扩展为支持 `Buffer` 或 Node stream 的通用上传函数，使用 `dataLength` 设置 `Content-Length` 并计算上传进度。
4. `uploadOpenListFile` 不再通过 `fs.readFileSync(localFilePath)` 把本地文件整块读入内存，改为 `fs.statSync` 获取大小并用 `fs.createReadStream` 流式上传。
